```python
# 규칙 1: gold의 단일문자/숫자 수식을 "라벨"로 분리 (정합 평가에서 제외)
import re
LABEL_PAT = re.compile(r'^\s*(?:[A-E]|[0-9]{1,2}|[①-⑩])\s*$')

def is_label_only(hwp_eq: str) -> bool:
    # 백틱·공백 제거 후 단일 보기/배점/번호인지 판정
    core = hwp_eq.replace('`', '').strip()
    return bool(LABEL_PAT.match(core))
```

```python
# 규칙 2: LaTeX → HWP 수식 1차 정규화 (백슬래시 제거 사전)
LATEX_TO_HWP = {
    r'\leq': 'le', r'\geq': 'ge', r'\neq': 'ne',
    r'\alpha': 'alpha', r'\beta': 'beta', r'\theta': 'theta',
    r'\left|': 'left|', r'\right|': 'right|',
    r'\left(': 'left(', r'\right)': 'right)',
    r'\cdot': 'cdot', r'\times': 'times',
}
def latex_to_hwp(s: str) -> str:
    s = re.sub(r'\\frac\{([^{}]+)\}\{([^{}]+)\}', r'{\1} over {\2}', s)
    for k, v in LATEX_TO_HWP.items():
        s = s.replace(k, v)
    return s
```

```python
# 규칙 3: 연립방정식 array 블록을 행 단위로 분해
def split_array_block(latex: str) -> list[str]:
    m = re.search(r'\\begin\{array\}\{[^}]*\}(.*?)\\end\{array\}',
                  latex, flags=re.DOTALL)
    if not m: return [latex]
    rows = [r.strip() for r in m.group(1).split(r'\\') if r.strip()]
    return rows  # 각 행을 개별 수식으로 비교
```

## A. LaTeX→HWP 변환 패턴
# LaTeX → HWP 수식 변환 패턴 분석

## 1. 핵심 관찰

### 1.1 데이터 정렬(alignment) 문제
입력 쌍을 보면 **raw LaTeX 개수와 gold HWP 개수가 일치하지 않는 경우가 다수**다.

| 시험지 | raw 개수 | gold 개수 | 비고 |
|---|---|---|---|
| 경신여고 #16 | 1 | 5 | gold는 보기(A,B,C,D,E) — OCR 누락 |
| 고려고 #1 | 1 | 3 | gold는 번호(1,2,3점) |
| 광덕고 #16 | 3 | 5 | gold는 점수/번호 |
| 금호고 #5 | 1 | 4 | gold는 보기 번호 |

→ **gold에 자주 등장하는 `숫자\``** (예: `3\``, `1\``, `10\``, `30\``)는 **수식 본문이 아니라 문제 배점/보기 번호 표기**이며, LaTeX raw에는 거의 존재하지 않는다.

### 1.2 실제 수식 변환이 성립하는 사례
정렬이 맞는 부분만 추출:

| raw LaTeX | gold HWP |
|---|---|
| `y=x^{2}+5 x` | `y=x^{2}+5x\`` |
| `y` | `y\`` |
| `x^{3}+x-2=0` | `x^{3}+x-2=0\`` |
| `2 \times 2` | `2 times 2\`` |
| `(i, j)` 관련 → `a_{i j}` | `a_{ij}\`` |
| `a_{i j}=2 i-j+3` | `a_{28}=2i-j+3\`` (OCR 오류) |
| `x` | `x\`` |
| `x^{3}-(a^{2}+1)x^{2}+(a^{2}-2a)x+2a=0` | 동일 구조 + `left/right` |
| `a` | `a\`` |

## 2. 변환 규칙표 (LaTeX → HWP)

| # | LaTeX 패턴 | HWP 변환 | 비고 |
|---|---|---|---|
| R1 | 식 끝 | 끝에 **백틱 `\``** 부착 | 단순 변수도 `y\``, `x\`` |
| R2 | `\\` (백슬래시 명령) | 백슬래시 제거, 명령어만 | `\times` → `times`, `\alpha` → `alpha` |
| R3 | `{i j}`, `{5 x}` 등 토큰 내부 공백 | **공백 제거** | `a_{i j}` → `a_{ij}`, `5 x` → `5x` |
| R4 | `\left(...\right)` | `left(...right)` (백슬래시만 제거) | 광덕고 동아여고 사례 |
| R5 | `^{...}`, `_{...}` 중괄호 첨자 | **그대로 유지** | `x^{2}`, `a_{ij}` |
| R6 | 식 사이 일반 공백 | 보존되거나 제거 | 일관성 없음 (R3은 제거, `2 times 2`는 유지) |
| R7 | 다중 수식(`\quad`, 콤마 구분) | **각각 분리된 HWP 항목**으로 변환 | gold가 리스트 형태 |

---

## 3. 불규칙 / 복잡 사례

### 3.1 OCR 오류로 인한 불규칙
- `a_{i j}=2 i-j+3` → `a_{28}=2i-j+3\`` : **`ij`가 `28`로 잘못 변환** (OCR 또는 매칭 오류). 규칙화 불가.
- `6^{\circ} \&`, `10 \frac{0000 \mathrm{r}^{m}}{...}` : 광주여고 raw에 OCR 노이즈가 섞여 있어 gold에 매핑되지 않음.

### 3.2 누락 / 추가
- 동명고 #7: raw에 있던 `\Lambda`(=Λ)가 gold에서는 `A`로 변환 (대문자 람다 → A 오인).
- gold에 등장하는 `1\``, `2\``, `10\``, `30\`` 등 **번호/배점 토큰**은 raw 수식과 무관 → 별도 추출 경로 필요.

### 3.3 구분자 처리
- 동아여고: raw `a^{2}-2 a \quad 2 a` (두 식이 `\quad`로 분리) → gold에는 하나의 식 안에 포함되거나 분할.
- 다중 LaTeX 항목이 하나의 HWP 식으로 합쳐지거나, 반대 경우도 존재.

---

## 4. 자동화 우선순위 권고

### Priority 1 (즉시 적용 — 높은 신뢰도)
1. **백슬래시 명령 변환**: `\times→times`, `\alpha→alpha`, `\beta→beta`, `\circ→circ`, `\quad→` (분리자로 사용), `\angle→angle`, `\frac→frac`, `\mathrm→rm`.
2. **식 종료에 `\`` 부착**: 모든 수식 HWP 출력 끝.
3. **`\left( \right)` → `left( right)`**.
4. **첨자 내부 공백 제거**: 정규식 `\{([^}]*)\}` 내부의 ` `를 제거하거나, `_{i j}` → `_{ij}`.

### Priority 2 (검증 후 적용)
5. **단일 변수도 그대로 wrap**: `x` → `x\``.
6. **`\quad`, 콤마 등 다중 수식 split 로직**: raw에서 식 단위 분리 → 각각 HWP 변환.
7. **숫자×변수 공백 제거**: `5 x` → `5x`, `2 a` → `2a` (단, `2 times 2`처럼 명령어 주변 공백은 보존).

### Priority 3 (보정/필터링 단계)
8. **gold의 "번호/배점 토큰" 분리**: `^\d+\`$` 패턴은 수식 변환 대상에서 **제외**하고 메타데이터로 처리.
9. **OCR 노이즈 필터**: `0000`, 단독 `\&`, 비정상 `\mathrm{r}^{m}` 등은 사전 정제.
10. **정렬(alignment) 모듈**: raw N개와 gold M개 매칭 시 길이·문자유사도 기반 매칭. 매칭되지 않는 gold 항목은 별도 분류.

### Priority 4 (장기 개선)
11. **OCR 후처리 사전**: `\Lambda↔A`, `ij↔28` 같은 빈번한 오인식 패턴을 학습 데이터로 축적.
12. **다중 수식 병합/분리 휴리스틱**: HWP의 한 항목이 LaTeX의 몇 개와 대응하는지 학습.

## B. gold 수식 >> raw (OCR 누락)
# Gold HWP가 Raw LaTeX보다 수식이 많은 이유 분석

## 1. Gold가 수식을 더 많이 가지는 근본 원인

### 1.1 HWP 수식 객체 카운팅 방식의 특성
HWP는 **문자 단독(예: `A`, `B`, `1`, `2`)도 별도 수식 객체**로 저장하는 경향이 강하다. 반면 raw LaTeX OCR은 본문 텍스트에 인라인된 단일 문자를 일반 텍스트로 인식하거나 아예 누락한다.

- **예시 (경신여고 #16)**: gold는 `{rm{A}}it` ~ `{rm{E}}it` 5개를 개별 수식으로 보유 → 본문에서 "다섯 학생 A, B, C, D, E"를 HWP에서는 각 알파벳을 수식으로 입력한 것.
- raw에서는 `5: x=-10-10` 같은 엉뚱한 OCR 결과 1개만 잡힘.

### 1.2 표(table) 내부 셀의 숫자/기호가 모두 개별 수식
확률·경우의수·통계 단원에서 자주 등장하는 **도수분포표/확률표**의 각 셀이 gold에는 수식으로 들어간다.

- **광덕고 #21**: raw에는 수식 0개 → 표 셀 `1`, `2`, `25`, `1`, `200`이 전부 누락.
- **국제고 #11**: 마찬가지로 raw 0개, gold는 `6`, `0`, `1`, `2`, `3` (보기 번호 또는 표 값) 보유.

### 1.3 보기 ㄱ/ㄴ/ㄷ 안의 수식
보기 박스 내부 명제식이 gold에는 잡히지만 raw에는 박스 텍스트만 OCR되고 내부 수식이 빠지는 경우.

- **명진고 #20**: gold에 `f left(x right)`, `g left(x right)`, `f left(x right) ge 2` 등 보기 내부 함수식이 별도 수식.
- raw는 `x, y, z`, `(x, y, z)`처럼 좌표 변수만 가져오고 보기 명제식은 누락.

### 1.4 집합/사건 기호의 분할 저장
사건 `A`, `B`가 HWP에서는 **개별 수식 객체 + 여백(`` ` ``)** 형태로 저장되는데, raw는 `\mathrm{A}, \mathrm{B}`처럼 하나로 묶어 OCR.

- **광덕고 #13, 광주고 #17**: gold에 `{rm{A}it}``, `{rm{B}it}``, ... 각각 분리 / raw는 `\mathrm{A}, \mathrm{B}` 단일.

### 1.5 단위·비율(%)·수치 답 보기
선택지·답안 후보 숫자(`1`, `2`, `15`, `200` 등)와 % 수치가 gold에는 수식, raw에는 문장 텍스트.

- **대동고 #9**: gold `5`, `3`, `15`, `1`, `15` (선택지 답안 후보) → raw는 본문 변수 표현만, 선택지 수치 누락.
- **광주고 #15**: gold `16`, `1`, `2` 등 선택지 정답 후보 / raw는 본문 수열 `1,6,11`, `5,10,14`만.

| 카테고리 | 누락 패턴 | 사례 |
|---|---|---|
| **A. 단일 알파벳(사건/점/집합)** | 본문 인라인 `A`, `B`, `C`를 텍스트화 또는 누락 | 경신여고 #16, 광덕고 #13 |
| **B. 표 셀 숫자** | 표 안의 도수/확률/계급값이 전부 누락 | 광덕고 #21, 국제고 #11 |

| **F. 짧은 정수 답(`-1`, `1`)** | 단답형 정답 수치가 본문 텍스트로 흡수 | 명진고 #20 |

## 3. 파싱 개선 방향 제안

### 3.1 OCR 후처리 단계 강화
1. **단일 문자 수식 복원기**
   - 정규식: 문장 중간의 고립된 라틴 대문자(`\b[A-Z]\b`)가 "사건 A", "점 B", "집합 C" 패턴 뒤에 오면 강제로 `\mathrm{X}` 수식으로 승격.
   - 한국어 트리거 단어: `사건`, `점`, `집합`, `함수`, `수열`, `벡터` 뒤 1~3글자 라틴 대문자 시퀀스.

2. **표(table) 셀 전수 수식화**
   - PDF 표 영역 검출(라인/그리드 또는 LayoutLM) 후, 각 셀의 숫자·기호를 **셀 단위로 개별 수식 토큰**으로 저장.
   - 표 본문 텍스트에서 추출되지 않은 셀이 있는지 gold와 카운트 매칭 검증.

3. **선택지 영역 분리 파서**
   - `①②③④⑤` 또는 `(1)(2)(3)(4)(5)` 마커를 기준으로 5개 토큰을 분할 → 각 토큰을 강제로 수식 객체로 등록.
   - 짧은 정수/분수/근호는 모두 수식으로 승격.

4. **보기 박스(ㄱ/ㄴ/ㄷ) 내부 재OCR**
   - 박스 영역을 별도 크롭하여 수식 OCR을 2차 적용. 박스 안의 `f(x)`, 부등식, 조건문은 손상 위험이 크므로 별도 모델 사용 권장.

### 3.2 정규화 규칙 (raw → gold 매핑)
- `\mathrm{A}, \mathrm{B}` → `{rm{A}}it`, `{rm{B}}it` 두 개로 split.
- 선택지 라인 `① 70 \%  ② 30 \%` → `[70%, 30%]` 각각 수식.
- 본문 "사건 A와 사건 B" → `A`, `B` 두 수식 추출.

### 3.3 품질 검증 지표 추가
- **수식 카운트 차이 임계값 알람**: `|gold_count - raw_count| ≥ 3`이면 강제 재검수 큐로 이동.
- **카테고리별 누락율 대시보드**: 표/선택지/보기/단일문자 4축으로 누락율 추적.

### 3.4 우선 적용 순위
1. **표 셀 추출** (누락량 최대, 광덕고 #21·국제고 #11처럼 raw=0인 케이스 다수)
2. **선택지 5개 강제 분할** (대동고·광주고 패턴)
3. **단일 알파벳 사건/점 승격** (확률·기하 단원 빈출)
4. **보기 박스 재OCR** (구현 난이도 높음, 후순위)

이 네 단계만 적용해도 위 8개 사례 중 6개 이상에서 수식 카운트가 gold와 근접할 것으로 예상된다.

## C. raw 수식 >> gold (OCR 과잉)
# Raw LaTeX 과잉 인식 패턴 분석

## 1. raw가 gold보다 많아지는 근본 원인

PDF OCR(Mathpix 등)은 페이지 내 **수식처럼 보이는 모든 영역**을 LaTeX로 변환하지만, gold HWP는 다음 원칙으로 작성된다:

- **그림/도형/표로 표현된 수식은 HWP에 옮기지 않는다** (별도 이미지로 처리)
- **연립방정식·행렬 등 복합 구조**는 사람이 분해해서 핵심 변수만 인라인으로 적는다
- **선택지(①~⑤)에 들어가는 단순 숫자/분수**는 수식 객체로 만들지 않는다
- **문제 본문 중 "구하시오" 직전의 결과식**만 수식으로 남기는 경향

---

## 2. OCR 과잉 수식화 패턴 목록

### 패턴 A. 연립방정식/연립부등식 `\left\{\begin{array}` 통째 인식
사람은 보통 그림 또는 별도 줄로 처리하지만 OCR은 한 덩어리 수식으로 추출.

- 금호고 #14: `\left\{\begin{array}{l}|x-a|<b \\ x^{2}-ax\geq x-a\end{array}\right.` → gold에는 없음 (그림/별도 박스)
- 동명고 #5: `\left\{\begin{array}{l}|x-1|\leq 3 \\ x^2-8x+15>0\end{array}\right.` → gold에서 제거
- 동신여고 #11: `\left\{\begin{array}{l}3x^2-2y^2+k=0 \\ ...\end{array}\right.` → 제거

### 패턴 B. 5지선다 보기(①~⑤)를 수식으로 추출
선택지는 HWP에서 별도 객체(번호 기호 + 짧은 식)로 처리되므로 raw의 보기 5개가 통째로 잉여가 됨.

- 경신여고 #12: `(4+α)(4+β)(4+γ)` 등 선택지 후보들
- 동명고 #5: `/5`, `-3≤x-1≤3` (이건 풀이 흔적까지 OCR됨)
- 대광여고 #5: `x=1,y=1`, `x=2,y=2` … (전형적인 선택지 5개)
- 동신여고 #11: `b_5`, `3x^2+k=2y^2` (선택지)

### 패턴 C. 행렬을 LaTeX `array`로 인식 vs HWP `pile/rpile`
gold는 행렬을 따로 안 적거나, 적더라도 HWP 수식 문법(`left( rpile{...} right)`)으로 변환. 변환 누락 시 raw 쪽이 그대로 남음.

- 광덕고 #12: 행렬 A, B → gold에 있긴 하나 HWP 문법으로 재작성
- 동성고 #19: 행렬 A → 동일 패턴

### 패턴 D. 풀이/해설 영역까지 OCR
문제 영역과 해설 영역의 구분이 안 되어 풀이 중간식이 raw에 포함.

- 경신여고 #12: `(2α+2β-γ)^2=-81` (조건식인지 풀이인지 모호)
- 동명고 #5: `-3≤x-1≤3` (명백한 풀이 과정)
- 동신여고 #11: `3x^2+k=2y^2` (변형 과정)

### 패턴 E. 단순 변수/상수까지 모두 수식 객체화
gold는 변수 하나(`a`, `b`, `x`)만 백틱(`)으로 처리하지만 raw는 모든 변수를 개별 수식으로 분리.

- 금호고 #14: raw `a`, `b` 각각 / gold도 비슷하나 raw에 잡힌 연립식이 사라짐
- 동신여고 #11: raw `x,y` (콤마 결합) vs gold `x`, `y`로 분리

### 패턴 F. OCR 환각(hallucination) — 존재하지 않는 기호
- 대광여고 #5: `\int x-y=0 x^2+y^2=50` — 적분기호 `\int`는 명백한 오인식(원래는 연립 중괄호일 가능성). gold가 비어있는 것도 문제 자체가 이미지 중심이거나 OCR이 완전히 실패했음을 시사.

### 패턴 G. 그리스 문자 나열을 한 수식으로 묶음
- 경신여고 #12: `α,β,γ` 한 덩어리. gold는 보통 각각 분리하거나 본문에서 한글 서술로 처리.

---

## 3. 파이프라인 개선 제안

### 3-1. 필터링 규칙 (raw에서 제거 후보)

| 규칙 | 매칭 패턴 | 처리 |
|---|---|---|
| R1 | `\left\{\s*\begin{array}` 포함 | **별도 "구조식" 태그**로 분류, 인라인 수식 목록에서 분리 |
| R2 | `\begin{array}` 안에 `&` 2개 이상 (행렬) | HWP `rpile/pile` 문법으로 **자동 변환** 후 비교 |
| R3 | 한 페이지에서 동일 변수 패턴이 5번 반복 (`x=1,y=1`, `x=2,y=2`…) | **선택지로 추정**, 본문 수식에서 제외 |
| R4 | `\int` 뒤에 `dx`/`d변수`가 없는 경우 | OCR 환각으로 간주, 경고 로그 |
| R5 | 길이 1~2자 단순 변수(`x`, `a`, `b`) | gold와 1:1 매칭 시 우선순위 낮춤 (개수 불일치 허용) |

### 3-2. 정규화 규칙

- **N1. 부등호/관계식 정규화**: `\leq` → `le`, `\geq` → `ge`, `<` → `&lt;`, `>` → `&gt;` (HWP 변환 일치)
- **N2. 곱셈 공백 제거**: `A B` → `AB`, `2 x^2` → `2x^2` (gold는 공백 거의 없음: `2x-3=0`)
- **N3. 행렬 변환**: `\begin{array}{cc} a & b \\ c & d \end{array}` → `rpile{ a && b # c && d }` 자동 매퍼 구현
- **N4. 그리스 문자 명령어 유지**: `\alpha` → `alpha` (HWP 수식문법)
- **N5. 연속된 콤마 변수 분리**: `\alpha, \beta, \gamma` → `[α, β, γ]` 리스트로 분해해 매칭

### 3-3. 영역 분할(권장 최우선)

대부분의 누락은 **문제/선택지/해설** 영역 구분 실패에서 발생.

1. **레이아웃 분석 선행**: 문제 번호(①②③④⑤, 원숫자) 검출 → 선택지 박스 좌표 추출
2. 선택지 박스 내부 OCR 결과는 별도 필드(`choices[]`)에 저장, 본문 수식과 분리
3. "[해설]", "[풀이]" 헤더 이후 영역은 raw에서 제외
4. 그림 영역(이미지 박스) 내부 텍스트는 `figure_text[]`로 분리

### 3-4. 검증 메트릭 제안

- raw 수식 개수 / gold 수식 개수 비율이 **3 이상**이면 자동 플래그
- raw에 `array` 포함했는데 gold에 `rpile` 없으면 → 변환 누락 후보
- gold 빈 리스트(`[]`)인데 raw 비어있지 않으면 → **OCR 전면 재검토** 필요 (대광여고 #5 케이스)

---

## 4. 우선순위 권장

1. **(High)** 선택지 영역 분리 — 패턴 B가 가장 빈번
2. **(High)** `\left\{\begin{array}` 연립식 별도 처리 — 패턴 A
3. **(Mid)** 행렬 LaTeX↔HWP 자동 변환기 — 패턴 C
4. **(Mid)** 풀이 영역 컷오프 — 패턴 D
5. **(Low)** 환각 검출 휴리스틱 — 패턴 F (드물지만 치명적)

## D. 이미지 대체 패턴
# Raw LaTeX → Gold HWP 이미지 대체 패턴 분석

## 1. 이미지로 대체되는 내용의 공통 패턴

세 문제 모두 **"두 함수 `y=f(x), y=g(x)`의 그래프가 그림과 같을 때..."** 형식의 그래프 해석 문제이다.

### 1-1. 이미지로 대체되는 요소

| 유형 | Raw LaTeX 예시 | 설명 |
|------|----------------|------|
| **좌표평면 그래프** | (PDF에는 그림이 있으나 OCR이 인식 실패) | `y=f(x)`, `y=g(x)`의 곡선 그래프 자체 |
| **그래프 위 좌표 라벨** | `x \leq -2`, `x \geq 2`, `-6 \leq x \leq 6` | 그래프상의 정의역/구간 표시 |
| **그래프 위 점/문자 라벨** | `\beta`, `A`, 숫자 `9` | 그래프 안에 표기된 좌표/점 이름 |

→ Raw OCR이 **그림 안의 텍스트(라벨)를 LaTeX 수식으로 잘못 추출**하고 있다.  
→ Gold에서는 이 라벨들이 모두 **그림 이미지의 일부**로 흡수되어 별도 수식으로 존재하지 않는다.

### 1-2. 수식으로 유지되는 요소

| Raw | Gold | 비고 |
|-----|------|------|
| `y=f(x), y=g(x)` | `y=f left(x right)` × 2개 | 본문 인용 수식은 유지 |
| `f(x)-g(x) \leq 0` | `f left(x right)-g left(x right) le 0` | 조건/부등식 본문은 유지 |
| `\{f(x)\}^{2} > f(x)g(x)` | (유지) | 본문 수식은 유지 |

→ **문제 본문에서 언급되는 수식**은 그대로 LaTeX→HWP 변환됨.

---

## 2. 이미지 대체 필요 vs 수식 처리 가능 구분 기준

### 🖼️ 이미지 대체가 필요한 경우
1. **그래프 자체** (좌표축, 곡선)
2. **그래프 내부에 위치한 라벨**: 
   - 축 위 좌표값 (`-2`, `2`, `-6`, `6`)
   - 곡선 위 점 이름 (`A`, `\beta`)
   - 구간 표시 (`x \leq -2` 같이 그래프에 주석으로 적힌 것)
3. **도형, 표(특수 서식), 기하 그림**

### 📝 수식으로 처리 가능한 경우
1. **본문 문장 안의 수식**: "함수 `y=f(x)`에 대하여..."
2. **조건식/부등식**: "단, `f(x)-g(x) \leq 0`을 만족..."
3. **선택지 안의 수식**

### ⚠️ 판별 어려움 (Raw OCR이 혼동하는 영역)
- 그래프 라벨인지 본문 수식인지 위치 정보 없으면 구분 불가
- 예: `x \leq -2`가 본문 조건일 수도, 그래프 정의역 라벨일 수도 있음

---

## 3. 파이프라인 개선 권고

### 3-1. 그림 영역 사전 탐지 (최우선)
- PDF 단계에서 **figure/image bounding box 검출** 후, 해당 영역의 OCR 결과는 LaTeX 추출 대상에서 **제외**하고 그림 placeholder(`[FIG]`)로 치환.
- 도구: PyMuPDF의 `page.get_images()`, layout parser(LayoutLMv3, PubLayNet), 또는 그래프 영역 segmentation 모델.

### 3-2. "그림 신호" 키워드 기반 휴리스틱
문제 텍스트에 다음 표현이 있으면 **그림 존재 강제 가정**:
- `그림과 같이`, `그래프가 그림과 같을 때`, `다음 그림`, `[그림]`
- 이 경우 인접 영역의 짧은 수식 토큰(`A`, `\beta`, `9`, `x \leq -2` 등)은 그림 라벨로 분류하여 제거 후보로 표시.

### 3-3. 수식 필터링 규칙
Raw LaTeX 토큰별 분류기 도입:

| 길이/형태 | 분류 |
|-----------|------|
| 단일 문자(`x`, `A`, `\beta`) + 그림 인접 | **그림 라벨 → 제거** |
| 단일 숫자(`9`, `-2`) + 그림 인접 | **그림 라벨 → 제거** |
| 등호/부등호 포함하며 본문 문장 내부 | **본문 수식 → 유지** |
| `y=f(x)` 같은 함수 정의 표현 | **본문 수식 → 유지** |

### 3-4. Gold 매칭 기반 학습 데이터 구축
- 위 3개 예시처럼 raw에는 있지만 gold에 없는 수식 토큰들을 **"이미지로 흡수된 라벨"** 라벨로 자동 수집.
- 추후 분류기(또는 LLM 프롬프트) 학습 자료로 활용.

### 3-5. 검수 플래그
- 문제에 `그림과 같이`가 있는데 추출된 이미지가 0개이면 **수동 검수 플래그** 표시.
- 반대로 본문에 그림 언급 없는데 단일문자 LaTeX 토큰이 다수 있으면 OCR 오류 의심 플래그.

### 3-6. 즉시 적용 가능한 룰 (요약)
```
IF (문제텍스트 ∋ "그림과 같이" OR "그래프가 그림") THEN
    1. 본문 문장 외부의 짧은 LaTeX(≤4 토큰) 제거
    2. 본문 문장 내부 수식만 LaTeX→HWP 변환
    3. 그림 placeholder 삽입 → 이후 이미지 캡쳐 모듈 연결
```

이 규칙만으로도 위 3개 사례에서 raw의 잉여 토큰(`x \leq -2`, `\beta`, `A`, `9` 등)을 자동 제거 가능하다.

## E. 서술형 처리 패턴
# 서술형 문제 raw OCR vs Gold HWP 분석

## 1. 주요 차이점

### 1.1 수식 표기 문법 차이 (LaTeX → HWP 수식)

| 항목 | raw (LaTeX) | gold (HWP) |
|------|-------------|------------|
| 공백/구분자 | 없음 (`kpx`) | 그대로 붙여쓰기 (`kpx`) 또는 토큰 사이 공백 |
| 줄바꿈 끝 | 없음 | 백틱(`` ` ``)으로 종결 — 예: `x``, `k`` |
| 부등호 | `\leq`, `\geq` | `le`, `ge` (LaTeX 백슬래시 제거형) |
| 절댓값 | `\left| ... \right|` | `left|` ... `right|` (백슬래시 없음) |
| 함수 인자 괄호 | `f(x)` | `f left(x right)` |
| 그리스문자 | `\alpha`, `\beta` | `alpha`, `beta` (백슬래시 제거) |
| 분수 | `\frac{a}{b}` | (예시엔 없으나 HWP식 `a over b` 추정) |

### 1.2 raw OCR의 누락/오인식

- **#20번**: raw가 **완전히 비어 있음** (`[]`) — 그림/표/조건 박스가 많은 서술형은 OCR이 수식을 거의 못 잡음.
- **#18번 대동고**: raw에 `(\psi)_{2}` 같은 **무의미한 오인식**만 남음. 실제 본문 부등식(`x-4 ≤ |x-2| ≤ 3` 류)은 gold에서 새로 작성됨.
- **#19번 대동고**: raw는 연립부등식 1개만 포착, gold는 다항함수 `f(x), g(x)` 정의 등 **본문 전체 수식**을 새로 입력.

### 1.3 문항 구성요소 수의 불일치

- raw의 수식 개수 ≠ gold의 수식 개수 (대부분 gold가 더 많음).
- gold에는 raw에 없던 **숫자 정답/단답 값**(`6``, `1``, `2``)이 별도 토큰으로 들어 있음 → **부분답·소문항 답안 슬롯**으로 추정.
- 광주고 #20처럼 raw는 빈 배열인데 gold는 단답 5개(`1`,6`,6`,1`,2`) → gold가 **빈칸 채우기형 정답 묶음**을 담는 용도.

### 1.4 표기 정규화

- LaTeX의 `\\`(줄바꿈), `\left\{\begin{array}...\end{array}\right.` 같은 **환경 명령**은 gold에서 완전히 사라짐 → 사람이 의미만 보고 HWP 수식으로 재작성.
- 변수 단독(`x`)도 gold에서는 `x`` 처럼 백틱이 붙어 **수식 단위(인라인 수식 셀)** 임을 표시.

## 2. 서술형 자동 처리 시 주의사항

### 2.1 raw에 의존하지 말 것
- 서술형은 **조건 박스, 연립식, 그림, 다단 수식**이 많아 OCR 누락률이 매우 높음 (#20처럼 0개도 발생).
- raw가 비어 있거나 1~2개만 잡혀도 정상 케이스로 처리되도록 **fallback 경로** 필요.

### 2.2 LaTeX → HWP 수식 변환 규칙 테이블화
필수 치환 매핑:
```
\leq → le        \geq → ge
\alpha → alpha   \beta → beta   (모든 그리스문자 동일)
\left| → left|   \right| → right|
\left( → left(   \right) → right)
\frac{a}{b} → {a} over {b}
\\ (행바꿈) → # 또는 줄바꿈 토큰
\begin{array}…\end{array} → matrix/case 구조로 재구성
```
- 각 수식 토큰 끝에 **백틱(`` ` ``) 부착** 규칙 적용.

### 2.3 함수 호출 괄호 변환
- `f(x)`, `g(α)` 등은 단순 `(`,`)`이 아니라 **`left( ... right)`** 로 감싸야 HWP에서 정상 렌더.
- 정규식: `([a-zA-Z])\(([^)]+)\)` → `\1 left(\2 right)`

### 2.4 토큰 단위(슬롯) 분리
- gold는 **문항 본문 수식 + 정답/빈칸 값**이 한 리스트에 섞여 있음.
- 자동화 시 (a) 문제 본문 수식, (b) `<보기>`/조건 박스, (c) 정답란 숫자를 **별도 슬롯**으로 구분 필요.
- 단순 숫자(`1`, `6`)도 백틱 붙여 수식 토큰으로 통일.

### 2.5 OCR 노이즈 필터링
- `(\psi)_{2}` 같은 단발성·문맥 무관 토큰은 **신뢰도 낮음**으로 마킹하고 검수 큐로 보냄.
- raw 토큰 개수가 gold 예상치보다 현저히 적으면(예: 50% 미만) **수동 입력 모드**로 전환.

### 2.6 사람 입력 단계 보존
- 서술형은 raw→gold 매핑이 1:1로 결정되지 않으므로, **완전 자동 변환을 목표로 하지 말고** "OCR 초안 → HWP 수식 규칙 변환 → 사람 보정"의 3단 파이프라인을 유지.
- 특히 연립부등식·case 구조·그림 의존 조건은 자동화 제외 대상으로 분류.

### 2.7 검증 체크리스트
- [ ] 모든 수식 토큰 끝에 백틱 존재?
- [ ] `\` 잔존 명령어 0개?
- [ ] `(`, `)` 단독 사용 시 `left/right` 누락 여부?
- [ ] 그리스문자/관계연산자 백슬래시 제거 완료?
- [ ] 빈 raw 케이스에 대한 gold 슬롯 템플릿 생성 여부?
