## 2. 즉시 적용 가능한 간단 규칙 (Quick Wins)

```python
# 규칙 1: 그리스 문자·관계연산자 표기 변환 (LaTeX → HWP)
LATEX_TO_HWP = {
    r"\alpha": "alpha", r"\beta": "beta", r"\gamma": "gamma",
    r"\leq": "le", r"\geq": "ge", r"\neq": "ne",
    r"\times": "times", r"\cdot": "cdot",
    r"\left|": "left|", r"\right|": "right|",
}
def normalize_symbols(s: str) -> str:
    for k, v in LATEX_TO_HWP.items():
        s = s.replace(k, v)
    return s
```

```python
# 규칙 2: 수식 내부 토큰 공백 제거 + 변수 뒤 백틱 부착
import re
def hwp_spacing(expr: str) -> str:
    expr = re.sub(r"\s+", "", expr)                    # 공백 제거
    expr = re.sub(r"([a-zA-Z])(?=[,)\s]|$)", r"\1`", expr)  # 변수 뒤 `
    return expr
# "a, b" → "a`,b`"  ,  "3 k p x^{2}" → "3kpx^{2}"
```

```python
# 규칙 3: 영역 태그별 수식 분리 (선택지/조건박스 제외)
EXCLUDE_TAGS = ("choice", "condition_box", "bogi", "figure_label")
def filter_main_body(items: list[dict]) -> list[dict]:
    return [x for x in items if x.get("region") not in EXCLUDE_TAGS]
# raw 5개 → 본문 2개로 축소되어 gold와 정합
```

## A. LaTeX→HWP 변환 패턴
# LaTeX(raw) → HWP 수식 변환 패턴 분석

## 1. 핵심 관찰

### 1.1 일대일 매핑이 성립하지 않음
- raw LaTeX 항목 수와 gold HWP 항목 수가 **자주 다르다**.
  - 예: `광주고 #1`은 raw 1개 / gold 0개
  - 예: `금호중앙여고 #6`은 raw에 부등식 시스템·`x`만 있는데, gold는 행렬·이차방정식 위주
  - 예: `광주제일고 #8`은 raw 5개 → gold 2개 (`\alpha+\beta` 항목 누락)
- **OCR이 본문에 등장한 모든 수식을 다 잡지 못하거나(누락), 반대로 HWP 정답이 본문 일부만 수식으로 처리**한 결과로 추정.

### 1.2 단순 변수/식의 변환 규칙은 비교적 명확
- 단일 변수(`a`, `b`, `x`)는 그대로 두되 **뒤에 backtick(`` ` ``)**을 붙임 → HWP에서 공백/정렬용 토큰.
- 예: `a, b` (raw) → `a`, `` b` `` (gold, 콤마 기준 분리 + ` 추가)

### 1.3 함수/괄호 변환
- `f(a, b)` → `f left(a,``b right)``` 
  - `(` → `left(`, `)` → `right)`
  - 콤마 뒤에 ` `` ` (백틱 2개 = 강제 공백) 삽입
  - 식 끝에 `` ` `` 추가

### 1.4 그리스 문자
- `\alpha`, `\beta` → ` alpha`, ` beta` (백슬래시 제거, 앞뒤 공백)
  - 예: `x=\alpha, y=\beta` → `x= alpha`, `` y= beta` ``

### 1.5 로마체(Roman) 변환
- 본문의 점 라벨 `A, B, C, D, E` 같은 알파벳을 HWP에서 `{rm{A}}it` 또는 `{rm{A}it}` 형태로 변환
- 두 형태(`{rm{A}}it` vs `{rm{A}it}`)가 혼재 → **표기 불일치(오타/스타일 차)**
- raw가 콤마로 묶인 한 줄("A, B, C, D, E, F, G")을 gold는 **각 글자별 항목으로 분리**

### 1.6 행렬/케이스(시스템) — 핵심 난점
- `\left\{\begin{array}{l} ... \\ ... \end{array}\right.` (연립부등식/연립방정식)
  - gold에서는 **시스템 통째로는 거의 나타나지 않음**. 대신 풀이 결과인 개별 식(`x= alpha`, `y= beta`)으로 분해.
- `\left(\begin{array}{lll} ... \end{array}\right)` (행렬)
  - gold에서 `left( rpile{ ... # ... } right)` 형태로 매핑 가능 (금호중앙여고 사례 참조).
  - `\\`(행 구분) → `#`
  - `&`(열 구분) → `&&` (HWP escape: `&amp;&amp;`)
  - `\begin{array}` 래퍼 → `rpile{ ... }` (열 1개) 또는 `matrix{ ... }` 류

### 1.7 부등식/절댓값
- `|x+2|<3`, `|x-a|+|x+1| \leq b` 같은 raw가 gold에는 **거의 등장하지 않음** → 본문 그대로 노출되거나 HWP 변환 대상에서 제외됨.

## 2. 변환 규칙 표 (LaTeX → HWP)

| # | LaTeX 패턴 | HWP 변환 | 비고 |
|---|---|---|---|
| 1 | 단일 변수 `a`, `x` | `a`, `` x` `` | 토큰 뒤 backtick 1개 |
| 2 | 콤마 나열 `a, b` | 항목 분리 후 각각 `` `a` ``, `` `b` `` | 항목 단위 분해 |
| 3 | `\alpha`, `\beta`, `\gamma` … | ` alpha`, ` beta`, ` gamma` | `\` 제거 + 공백 1칸 |
| 4 | `\leq`, `\geq` | `<=`, `>=` (HWP 표준) | 표본에 직접 등장 없음 |
| 5 | `f(a,b)` | `f left(a,``b right)`` ` | `(`→`left(`, `)`→`right)`, `,`→`,``` |
| 6 | `x^{2}` | `x^{2}` | 거의 동일 (중괄호 유지) |
| 7 | 분수 `\frac{a}{b}` | `{a} over {b}` 또는 `a over b` | 표본 직접 없음, 일반 규칙 |
| 8 | `\begin{array}{l} A \\ B \end{array}` (연립) | **분해됨**: 결과식만 개별 항목 | 자동화 시 OCR 단계에서 분해 필요 |
| 9 | `\begin{array}{ll} a&b\\c&d \end{array}` (행렬) | `left( rpile{ a && b # c && d } right)` | `&`→`&&`(=`&amp;&amp;`), `\\`→`#` |
| 10 | 알파벳 라벨 `A, B, C` | `{rm{A}}it`, `{rm{B}}it`, … | Roman 표기, 항목별 분리 |
| 11 | `\left\{` / `\right.` | HWP에서는 보통 케이스 분해 후 사라짐 | 시스템 통째 변환 회피 |

## 3. 복잡/불규칙 사례

### 3.1 Gold가 raw에 없는 식을 포함 (금호중앙여고 #6)
- raw: 연립부등식, `x`
- gold: `x^2-2x+2=0`, 두 개의 행렬 `A`, `B`, `BA`
- → **OCR 누락이 매우 큼**. 본문 행렬·이차방정식이 OCR 단계에서 빠진 것으로 보임.

### 3.2 Roman 표기 두 가지 (경신여고 vs 대동고)
- 경신여고: `{rm{A}}it`
- 대동고: `{rm{A}it}`
- → **사람 입력 변동성**. 정규화 단계에서 한 형태로 통일 필요 (`{rm A}` 권장).

### 3.3 항목 단위 정의 불일치
- raw는 "한 줄에 콤마로 나열" / gold는 "글자 단위 분리" → tokenization 기준이 다름.
- 예: `A, B, C, D, E, F, G` (raw 1개) → gold 5개(`A`~`E`만, F·G 누락)

### 3.4 시스템(연립) → 결과식 치환
- raw의 `\left\{...\right.` 형태는 gold에서 거의 사라지고, 해당 문제의 **결론 변수**(`x=α`, `y=β`)만 남음.
- → 단순 문자열 변환이 아닌 **문제 의미 해석 후 항목 재구성**이 필요.

### 3.5 광주고 #1, 동성고 #5
- raw에는 식이 있는데 gold는 거의 비어있거나 `` x` `` 한 개. 
- → gold가 **본문의 일부 단어 강조용 수식**만 추출하는 정책일 가능성.

## 4. 자동화 권고 (우선순위)

### P0 — 데이터 정합성 진단 먼저
1. **항목 개수 매칭 정책 정의**
   - raw·gold 항목 수가 다른 경우 다수 → 단순 zip 매칭으로 학습/평가하면 노이즈가 큼.
   - 문제 본문 텍스트 기준으로 "raw 토큰 ↔ gold 토큰" 정렬 알고리즘(LCS 또는 유사도 기반) 필요.
2. **OCR 누락 점검**: 특히 행렬·연립식·각주(부호식)가 빈번히 빠짐. PDF 영역 검증 단계 추가.

### P1 — 결정적(deterministic) 치환 규칙 구현
다음 규칙은 충돌이 거의 없으므로 즉시 코드화:
- `\alpha|\beta|\gamma|...` → ` alpha| beta| gamma|...`
- `\left(` → `left(`, `\right)` → `right)`
- `\leq` → `<=`, `\geq` → `>=`, `\neq` → `!=`
- `\frac{X}{Y}` → `{X} over {Y}`
- 단일 토큰 끝에 `` ` `` 추가 (단, 등호·괄호 끝일 때만)

### P2 — 구조 변환 규칙
- `\begin{array}` + `\\` + `&` → `matrix{}` / `rpile{}` 변환기 작성
  - 행 구분 `\\` → `#`
  - 열 구분 `&` → `&&`
  - 좌우 괄호 종류(`\left(`, `\left[`, `\left\{`)에 따라 wrapper 선택
- `\left\{ ... \\ ... \right.` (cases) → `cases{ A # B }` 또는 분해

### P3 — 정규화 사전(Roman, 텍스트 라벨)
- 대문자 라벨 `A,B,C,...` 자동 인식 → `{rm A}` 형식 통일
- 두 변형(`{rm{A}}it`, `{rm{A}it}`) 모두 입력으로 받아 한 가지 출력으로 통일

### P4 — 의미 기반 분해(고난도)
- 연립방정식 → "해" 항목 추출은 LLM/룰 기반 후처리로만 가능.
- 자동화 1차 버전에서는 **시스템은 시스템 그대로 변환**, gold가 분해 형태면 평가 시 매칭 룰을 느슨하게.

### P5 — 평가 메트릭
- 정확 일치(EM)는 표본 변동성(백틱 위치, 공백 수) 때문에 너무 엄격함.
- **정규화 후 비교**(연속 공백 1개로 축약, 후행 backtick 제거, `{rm{A}}it`↔`{rm A}` 통일) 후 EM 측정 권장.

## 5. 요약
- gold HWP는 **모든 수식의 1:1 변환이 아니라 "출제자가 식으로 표기한 핵심 토큰"만 담는 경향**이 강함.
- 결정적 치환(그리스 문자, `\left/\right`, 분수, 부등호)부터 코드화하면 ROI가 가장 높음.
- 행렬/연립은 구조 파서 필요. 연립의 "해" 형태 분해는 후처리 단계로 분리할 것.
- **데이터 품질 진단(누락·중복·라벨 불일치)이 자동화보다 선행**되어야 함.

## B. gold 수식 >> raw (OCR 누락)
# Gold HWP 수식 과다 발생 분석 보고서

## 1. Gold에서 수식이 더 많은 이유

### 1.1 단일 문자/기호의 개별 수식화
HWP에서는 본문 중 **알파벳 변수 하나하나**가 별도의 수식 객체로 삽입된다.

- 예시 (`경신여고 #16`): `A`, `B`, `C`, `D`, `E` 다섯 개의 점(또는 선택지)이 각각 `{rm{A}it}`, `{rm{B}it}` … 처럼 **개별 수식**으로 등록됨.
- raw LaTeX는 이를 본문 텍스트(혹은 단일 `5: x=-10-10`처럼 노이즈)로 처리하거나 아예 누락.

### 1.2 변수 + 조사/공백의 분리 수식화
HWP는 `A는`, `B와` 같은 표현에서 **알파벳만 수식**, 한글 조사는 일반 텍스트로 분리한다. 그래서 한 문장 안에서 수식 객체가 여러 개로 늘어난다.

- 예시 (`광덕고 #13`): raw는 `\mathrm{A}, \mathrm{B}`처럼 하나의 수식으로 합쳐 OCR했지만, gold는 `{rm{A}it}`, `{rm{B}it}` 두 개로 분리.

### 1.3 표·확률값의 셀 단위 수식화
표 내부 수치(%, 분수, 변수)는 **셀마다 독립 수식**이다.

- 예시 (`광덕고 #13`): `60%`, `40%`, `70%`, `30%` 각각이 별도 수식. raw에서는 일부(`60 \%`)만 인식되거나 텍스트로 흡수됨.

### 1.4 보기(선택지) 내 수식 분할
①~⑤ 보기에서 각 항목의 수식이 개별 객체로 들어가는데, raw OCR은 선택지 자체를 인식하지 못하는 경우가 많다.

## 2. raw OCR이 놓치고 있는 유형 (카테고리별)

| 카테고리 | 설명 | 사례 |
|---|---|---|
| **A. 단일 변수 문자** | 본문에 흩어진 `A`, `B`, `C`… 등을 텍스트로 흡수 | 경신여고 #16의 점 5개 |
| **B. 표 셀 내 수치 수식** | 표 안의 `60%`, `40%` 같은 셀이 누락 | 광덕고 #13의 70%, 30% |
| **C. 변수 + 조사 분리** | `A와 B`를 하나의 수식으로 묶거나 통째로 텍스트화 | `\mathrm{A}, \mathrm{B}` vs A·B 분리 |
| **D. 보기(①②③④⑤) 내 수식** | 선택지 영역 OCR 자체가 실패 | 광덕고 #13의 ④⑤ 손실 |
| **E. 점/도형 라벨** | 기하 문제의 점 라벨이 이미지로 처리되어 누락 | 경신여고 #16의 A~E 점 라벨 |
| **F. 서체 정보(it/rm) 손실** | gold는 `rm`, `it`로 정자/이탤릭 구분, raw는 `\mathrm`만 사용하거나 누락 | `{rm{A}it}` 구조 |

---

## 3. 파싱 개선 방향 제안

### 3.1 OCR 단계
1. **표 OCR 강화**: PDF 내 표 영역을 별도 검출 후 셀별 텍스트/수식 분리 파이프라인 적용. 표 셀 안의 모든 숫자·기호를 수식 후보로 등록.
2. **단일 문자 변수 인식**: 본문 내 문장 중간의 고립된 알파벳(`A`, `B` 등)을 텍스트로 합치지 말고 **수식 토큰으로 강제 승격**. 정규식: `(?<![A-Za-z])[A-Z](?=[는은이가와과의를을,．\s])`.
3. **보기(①~⑤) 영역 검출**: 동그라미 숫자 직후 컨텐츠를 별도 OCR pass로 처리, 누락 방지.
4. **점/라벨 OCR**: 기하 이미지 내 라벨은 vision 모델에서 별도 처리해 라벨 리스트만 수식으로 export.

### 3.2 정규화 단계 (raw → gold 매핑)
1. **수식 분리(splitter)**: `\mathrm{A}, \mathrm{B}` → `[\mathrm{A}, \mathrm{B}]` 두 개로 split. 콤마·공백·`그리고`·`및`·`와` 등을 separator로 사용.
2. **서체 정규화 매핑**:
   - raw `\mathrm{X}` ↔ gold `{rm{X}it}` (HWP 특유 표기)
   - `\%` ↔ ` ``% ``` 등 공백·백틱 패턴 학습
3. **단위·기호 보존**: `%`, `°`, `cm` 등이 수식 안에 들어가는지 본문에 들어가는지 결정하는 규칙을 gold 통계로 학습.

### 3.3 검증 단계
1. **수식 개수 diff 모니터링**: 문제별 raw vs gold 수식 개수 차이가 임계값(예: 3개) 이상이면 자동 플래그.
2. **카테고리 분류기**: 누락 유형(A~F)을 자동 라벨링해 어느 유형이 가장 손실 큰지 대시보드화.
3. **gold 패턴 사전 구축**: `{rm{·}it}`, ` ``% `` ` 같은 HWP 특수 표기를 사전(dict)으로 정리해 raw→gold 변환 후 비교가 가능하도록 통일.

### 3.4 우선순위
1. (가장 임팩트 큼) **표 셀 단위 수식화** + **단일 변수 승격**
2. **보기 영역 별도 OCR**
3. **서체/공백 정규화 사전화**

## C. raw 수식 >> gold (OCR 과잉)
# Raw LaTeX vs Gold HWP 수식 개수 불일치 분석

## 1. 핵심 원인 진단

OCR(raw)은 PDF에서 **모든 수식 영역을 LaTeX로 추출**하지만, gold HWP는 **문항의 본문에 실제로 등장하는 미지수/조건만 수식으로 입력**하고 나머지는 **(가), (나), (다) 박스나 보기/조건 영역의 수식**으로 처리한다. 즉:

- **조건 박스(가)·(나)·(다), 보기 ㄱ·ㄴ·ㄷ** 안의 수식은 HWP에서 별도 도형/표 안에 들어가 본문 수식 리스트에서 누락된 것처럼 보임
- **선택지 ①~⑤의 수식**도 HWP에서는 별도 처리되어 본문 gold 리스트에 포함되지 않음
- 결과적으로 raw는 5개 ↔ gold는 1~3개 형태의 비대칭이 발생

---

## 2. OCR이 과잉 수식화하는 패턴 목록

### 패턴 A: 선택지(①~⑤) 수식 과잉 추출
- 경신여고 #12: `(4+\alpha)(4+\beta)(4+\gamma)` 같은 선택지 항이 raw에 포함됨
- 금호중앙여고 #18: `\frac{1}{6}, \frac{1}{3}, \frac{1}{2}` → 5지선다 선택지값
- 대광여고 #21: `9<k \leq 12` → 선택지 항

### 패턴 B: 조건 박스/연립조건 중복 추출
- 동신여고 #11: `\left\{\begin{array}{l}3x^2-2y^2+k=0 \\ 2x^2+4xy+2y^2=2\end{array}\right.` 와 `3x^{2}+k=2y^{2}` 둘 다 추출
- 금호중앙여고 #18: 연립부등식 array 전체 + 선택지값 동시 추출
- 대광여고 #21: `\left\{ x^2-3x-18<0, |3x-9| \geq k \right.` 박스 추출

### 패턴 C: 동일 식의 변형/반복 등장
- 고려고 #19: 동일 방정식 `x^3+ax^2+bx+c=0`이 3회, `-cx^3+bx^2-ax+1=0`이 2회 → gold는 5개 모두 유지하나 raw와 1:1 매칭 안 됨 (이 경우는 오히려 정상)
- 광주제일고 #22: `\alpha, \beta, \gamma`가 3회 반복 추출 → gold는 1회로 통합 안 함, 아예 제거

### 패턴 D: 단일 변수/그리스문자 과잉 토큰화
- 경신여고 #12: `x`, `\alpha,\beta,\gamma` 등 단독 변수까지 별도 수식으로 분리
- 광주제일고 #22: `\alpha,\beta,\gamma`를 3번 별도 추출
- 대성여고 #11: `\alpha,\beta,\gamma,\delta(\alpha<\beta<\gamma<\delta)` 정의구문 통째로 추출

### 패턴 E: OCR 깨짐/노이즈 수식
- 금호중앙여고 #18: `6x^{2}-5x-4 \geq^{2} 0` (`\geq^2` 비정상)
- 동신여고 #11: `b_5` (문맥 무관한 추출, 실제 문항에 없을 가능성)
- 대성여고 #11: `\alpha^2+\beta^2+\gamma^4+\delta^4` (2제곱과 4제곱이 섞임 — OCR 오류)

### 패턴 F: 보기/추가 조건절 수식
- 대성여고 #11: `\frac{\delta-\alpha}{\gamma-\beta}=3` → 조건 (가) 박스 내용
- 동성고 #17: `(a,b)`, `A` → 보기에 등장하는 좌표쌍·집합 기호

---

## 3. 파이프라인 개선 제안

### 3-1. 영역 인식 기반 분리
- **선택지 영역(①~⑤)** 을 OCR 단계에서 별도 필드로 분리하여 본문 수식 리스트에서 제외
- **조건 박스(가),(나),(다) / 보기 ㄱ,ㄴ,ㄷ**도 별도 컨테이너로 추출
- gold HWP 구조와 동일하게 `본문 / 조건 / 선택지`로 슬롯 분리

### 3-2. 정규화 규칙
| 규칙 | 처리 |
|---|---|
| `\left\{\begin{array}...\end{array}\right.` | 연립조건 → 줄단위 분해 후 HWP 수식 문법(`{`,`}` 줄바꿈)으로 변환 |
| 동일 수식 중복 | 본문 위치 단위로만 dedup (전역 dedup은 금지 — 고려고 #19처럼 의도적 반복 존재) |
| 단독 변수 `x`, `k`, `a` | 유지하되 gold 비교 시 trailing backtick(`` ` ``) 정규화 |
| `\alpha,\beta,\gamma` 같은 나열 | HWP의 `alpha`, `beta`, `gamma` 토큰으로 매핑 |

### 3-3. 노이즈 필터링 휴리스틱
- `\geq^{2}`, `\leq^{2}` 처럼 **부등호 뒤 위첨자** → OCR 오류로 간주, 위첨자 제거
- `b_5`, `A`, `(a,b)` 같이 **문항 본문에서 한 번도 정의되지 않은 심볼** → 컨텍스트 사전과 대조 후 제거
- `\alpha^2+\beta^2+\gamma^4+\delta^4` 처럼 **지수 패턴 불균형** → 경고 플래그 후 사람 확인

### 3-4. raw↔gold 정렬 전처리
1. raw LaTeX → HWP 수식 문법 변환 (`\frac{a}{b}` → `{a} over {b}`, `\left(` → `left(` 등)
2. trailing backtick(`` ` ``) 정규화
3. 본문/조건/선택지 슬롯별로 매칭 후 길이 불일치 비율 계산
4. **raw 수식 수가 gold의 2배 이상**이면 선택지·조건 박스가 본문에 섞였다고 판정 → 재분할 트리거

### 3-5. 자동 품질 체크 지표
- `len(raw_eqs) / len(gold_eqs) > 2.0` → 과잉 추출 의심 케이스 자동 플래그
- gold에 없는 raw 수식 중 `①②③④⑤` 또는 선택지 패턴(`\frac{1}{n}`, 정수 부등식 등) 비율 측정
- 조건 박스 키워드(`(가)`, `(나)`, `\begin{array}`) 동반 여부로 분류

---

## 4. 우선 적용 권장

1. **선택지 분리**가 가장 큰 효과 (위 8개 사례 중 5개 해결)
2. **연립조건 array 분해 + HWP 문법 변환**으로 동신여고/금호중앙여고/대광여고 케이스 정상화
3. **단독 변수 trailing backtick 정규화**로 표면적 불일치 다수 제거

## D. 이미지 대체 패턴
# Raw LaTeX → Gold 이미지 대체 패턴 분석

## 1. 이미지로 대체되는 핵심 패턴

### 패턴 A: **그래프/도형 자체에 대한 OCR 잡음**
PDF에 그래프나 도형 이미지가 포함되어 있을 때, OCR이 그래프 내부의 좌표·라벨·눈금을 텍스트/수식으로 잘못 인식한다.

- **#18 광덕고**: `'\mathrm{C}=0'`, `'4 \times 1 \times 4320'`, `'L \neq 0+x 9 \times 9 \times 3 \times 3'`
  → 도형/표 내부 라벨이 깨져서 OCR된 전형적 사례. gold는 **빈 배열 `[]`** = 완전히 이미지로 대체.
- **#18 광주고**: `'1,2,3,4,5,6,7'` (벤다이어그램 내부 원소들), 집합 라벨 `A, B`
  → gold는 라벨 `A`, `B`만 수식으로 유지하고 나머지는 이미지화.

### 패턴 B: **함수 그래프 문제의 본문 vs 그림 분리**
"y=f(x), y=g(x)의 그래프가 다음과 같다" 형식의 문제에서:
- **본문 수식**(조건, 부등식 등)은 gold에서도 유지됨
- **그래프 내부 라벨**(`x ≤ -2`, `x ≥ 2`, `β`, `A` 같은 좌표/구간 표시)은 이미지로 흡수됨

| 문제 | gold 유지 (본문) | gold 제거 (그림 내부) |
|---|---|---|
| 국제고 #4 | `y=f(x)`, `y=g(x)`, `f(x)-g(x)≤0` | `x≤-2`, `x≥2` |
| 금호고 #7 | `y=f(x)`, `y=g(x)`, `x` | `{f(x)}²>f(x)g(x)`, `β`, `A` |
| 대동고 #11 | `y=f(x)`, `y=g(x)`, `-6≤x≤6`, `x` | `{f(x)}²>f(x)g(x)` |

> ⚠️ **불일치 주의**: 금호고는 `{f(x)}²>f(x)g(x)`가 그림 내부 라벨로 처리됐지만, 대동고는 동일 표현이 본문 수식으로 유지될 수도 있음 → 사람 작업자마다 기준이 다름.

---

## 2. 이미지 대체 필요 여부 판단 기준

### 🖼️ 이미지로 대체해야 하는 경우
1. **그래프 캔버스 내부의 좌표/구간 라벨** (예: `x≤-2`가 그래프 영역 안에 적혀 있음)
2. **벤다이어그램, 표, 수형도 내부의 원소·셀 값**
3. **OCR이 명백히 깨진 토큰** (예: `L \neq 0+x 9 \times 9 \times 3 \times 3`)
4. **위치 정보가 의미를 가지는 수식** (좌표평면 위 점, 화살표 옆 수식 등)

### ✏️ 수식으로 유지 가능한 경우
1. **본문 서술 내 함수 정의** (`y=f(x)`, `y=g(x)`)
2. **본문 조건식** (`f(x)-g(x)≤0`, `-6≤x≤6`)
3. **단일 변수/집합 라벨** (`A`, `B`, `x`) — 본문에서 언급될 때

---

## 3. 파이프라인 개선 권고

### (1) **레이아웃 기반 분리가 최우선**
OCR 단계에서 단순 텍스트 추출이 아니라 **bounding box 기반 영역 분류**가 필요:
- 그림/그래프 영역(figure bbox) 안에 있는 LaTeX는 **그림에 귀속**
- 본문 텍스트 라인의 LaTeX만 **수식 추출 대상**

권고: PDF → 이미지 영역 detector(LayoutParser, PubLayNet 등) → 영역별 OCR 라우팅.

### (2) **OCR 노이즈 필터**
`L \neq 0+x 9 \times 9 \times 3 \times 3` 같은 결과는 다음 휴리스틱으로 폐기:
- 의미 없는 변수 조합 + `\times` 반복
- 같은 수가 3회 이상 반복 곱셈
- 단일 LaTeX 내에 미정의 변수 4개 이상

### (3) **본문/그림 라벨 분류기**
LaTeX 토큰을 다음 두 클래스로 분류:
- `INLINE_BODY`: 본문 흐름에 포함 → HWP 수식 변환
- `FIGURE_LABEL`: 그림 내부 라벨 → 그림에 흡수, HWP 출력에서 제외

특징(feature) 예시: bbox 위치, 주변 한국어 텍스트 존재 여부, 같은 라인 내 서술어 유무.

### (4) **그래프 문제 템플릿화**
"y=f(x), y=g(x)의 그래프가 다음과 같다" 같은 정형 문장은 템플릿으로 감지하여:
- 본문 수식 화이트리스트: `y=f(x)`, `y=g(x)`, 부등식·집합 조건
- 그래프 내부 라벨은 자동으로 figure 영역으로 송부

### (5) **gold 변환 규칙 일관성 검증**
같은 표현이 어떤 문제에서는 본문 수식, 다른 문제에서는 이미지로 처리되는 케이스(예: `{f(x)}²>f(x)g(x)`)를 모아 **작업자 가이드라인 정비**가 필요. 현 데이터로는 학습 신호가 흔들림.

### (6) **빈 gold(`[]`) 케이스 처리**
gold가 완전히 빈 배열인 문제(#광덕고 18번)는 **"전체가 도형/표 문제"** 시그널 → 별도 플래그를 두고 raw LaTeX를 모두 폐기하는 분기 추가.

## 요약
> **핵심 규칙**: "본문에서 읽히는 수식은 LaTeX 유지, 그림 안에 박힌 수식은 이미지로 흡수." 이를 위해 **PDF 레이아웃 분석 → 영역별 OCR → 분류기 기반 LaTeX 필터링**의 3단계 파이프라인이 필요하다.

## E. 서술형 처리 패턴
# 서술형 문제 raw OCR vs Gold 작성 방식 분석

## 1. 주요 차이점

### 1.1 문법/표기 체계의 근본적 차이
- **raw**: 표준 LaTeX 문법 (`\alpha`, `\beta`, `\leq`, `\frac{}{}`, `\left\{`, `\begin{array}` 등)
- **gold**: HWP 수식 스크립트 문법
  - 그리스 문자: `\alpha` → `alpha` (백슬래시 제거, 공백 구분)
  - 관계연산자: `\leq` → `le`
  - 절댓값: `\left| ... \right|` → `left|` ... `right|` (공백·백틱 기반)
  - 분수/근호 등도 LaTeX 명령어 대신 HWP 키워드 사용

### 1.2 공백 처리 방식
- **raw**: 토큰 사이 공백 빈번 (`3 k p x^{2}`, `-4 k^{3}`)
- **gold**: 변수·계수 사이 공백 제거 (`3kpx^{2}`, `-4k^{3}`)
- **gold 특유**: 수식 끝마다 백틱(`` ` ``) 부착 → HWP 수식 종결자 역할

### 1.3 내용 자체의 차이 (OCR 누락/오인식)
- **대동고 #18**: raw는 `(\psi)_{2}` 단 하나만 추출 → OCR이 거의 실패한 문항. gold에는 부등식·변수 4개가 정상 존재.
- **대동고 #19**: raw에는 연립부등식이 추출됨. gold에는 전혀 다른 다항식 `f(x), g(x)` 관련 식이 존재 → **문제 본문과 (가)/(나)/조건부 박스의 매칭 불일치** 의심.
- **광주고 #19**: raw 5개 vs gold 5개지만 순서·내용 불일치 (`\alpha, \beta, \alpha>\beta` → `n, k, p`) → 다른 부분 수식을 추출한 것으로 보임.

### 1.4 구조 보존 차이
- **raw**: `array`, `cases` 등 LaTeX 환경 그대로 보존
- **gold**: HWP에서는 줄바꿈/박스 구조를 별도 처리, 수식 스크립트 내에는 단일 라인으로 단순화

### 1.5 데이터 양/순서 불일치
- raw가 비어있어도 gold에는 수식 존재 (광주고 #20은 양쪽 모두 빈 케이스 – 텍스트 위주 서술형 가능)
- raw·gold 모두 비어있는 경우 → 순수 텍스트 서술형 가능성

## 2. 자동 처리 시 주의 사항

### 2.1 변환 규칙(매핑) 필수 항목
| LaTeX (raw) | HWP (gold) |
|---|---|
| `\alpha`, `\beta`, `\gamma` | `alpha`, `beta`, `gamma` (앞뒤 공백) |
| `\leq`, `\geq`, `\neq` | `le`, `ge`, `ne` |
| `\left\|`, `\right\|` | `left|`, `right|` |
| `\frac{a}{b}` | `{a} over {b}` |
| `\sqrt{}` | `sqrt{}` |
| 토큰 사이 공백 | 변수·계수 결합 시 공백 제거 |
| 수식 끝 | 백틱 `` ` `` 부착 |

### 2.2 서술형 특유 리스크
1. **OCR 추출 실패율이 객관식보다 높음**  
   - 서술형은 박스/조건부/풀이공간 등 복잡한 레이아웃 포함 → raw가 1~2개만 추출되거나 빈 배열인 경우 빈번. 빈 raw에 대한 fallback 필요.

2. **수식 순서/매칭 어긋남**  
   - raw·gold 수식 개수가 같아도 같은 수식이 아닐 수 있음 → 단순 인덱스 매칭 금지. 의미 기반 정렬·검증 필요.

3. **(가)(나)(다) 박스 처리**  
   - 서술형은 조건 박스가 다중 라인으로 등장. raw가 `array` 환경으로 묶거나 일부만 캡처 → gold에서는 박스별로 분리되어 저장될 수 있음.

4. **공백·백틱 정규화 단계 분리**  
   - 변환을 ① 명령어 매핑 → ② 공백 정리 → ③ 백틱 부착 순으로 단계화. 한꺼번에 처리 시 토큰 경계 오류.

5. **빈 결과 보존**  
   - 양쪽 모두 빈 경우는 정상(텍스트 서술). 강제 채움 금지.

6. **부등호/절댓값 인식**  
   - `|x-2|` 같은 절댓값은 OCR에서 `1`, `l`, `I`와 혼동 가능 → 별도 검증 룰 적용.

7. **변수 결합 규칙 주의**  
   - `3 k p x` → `3kpx`처럼 결합하되, 함수명·연산자와는 결합 금지 (`f(x)`는 `fleft(xright)` 아님 → `f left(x right)`).

### 2.3 권장 파이프라인
1. raw LaTeX → HWP 스크립트 자동 변환기 통과
2. 변환 결과를 gold와 토큰 단위 diff
3. **diff가 임계치 이상 → 사람 검수 큐로 이관** (특히 서술형은 자동 통과 비율을 낮추고 검수 비중 ↑)
4. raw 빈 배열 + gold 비어있지 않음 케이스는 **OCR 재처리** 우선순위 부여
